# 计算机图形学第一次实验报告（草稿）

## 1. 实验目的

1. 掌握 OpenGL 图形程序的基本开发流程（初始化、着色器、顶点数据、渲染循环）。
2. 实现二维图形绘制与二维变换：平移、旋转、缩放、反射、切变。
3. 实现三维图形绘制与三维变换：平移、绕 X/Y/Z 轴旋转、缩放。
4. 实现二维与三维组合变换，并验证“变换顺序不同，结果不同”。
5. 实现视图变换与投影切换（透视/正交），理解相机与投影矩阵的作用。
6. 实现键盘交互控制，形成“可演示、可复现、可验证”的实验工程。

## 2. 实验环境

- 操作系统：Windows 10/11
- 开发工具：VS Code、CMake
- 编译器：MSVC（Visual Studio Build Tools）
- 编程语言：C++17
- 图形库：OpenGL 3.3 Core + GLFW + GLAD
- 构建方式：CMake 子目录工程（根目录 + src 子工程）

## 3. 实验原理

### 3.1 齐次坐标与矩阵约定

本实验统一采用 4x4 齐次矩阵描述二维/三维变换，工程内部矩阵为列主序存储：

$$
index = col \times 4 + row
$$

与 OpenGL `glUniformMatrix4fv(..., GL_FALSE, m)` 保持一致，避免转置引发的方向错误。

### 3.2 二维与三维基本变换

- 平移：通过平移矩阵改变图形位置。
- 旋转：二维围绕 Z 轴，三维支持绕 X/Y/Z 轴旋转。
- 缩放：沿各轴独立缩放。
- 反射与切变：二维中实现关于轴反射与切变，用于观察形状与方向变化。

### 3.3 MVP 管线

顶点坐标在着色器中按如下顺序变换：

$$
\mathbf{p}_{clip} = \mathbf{P} \cdot \mathbf{V} \cdot \mathbf{M} \cdot \mathbf{p}_{local}
$$

其中：
- $\mathbf{M}$：模型矩阵，控制物体局部变换。
- $\mathbf{V}$：视图矩阵，由 `lookAt(eye, center, up)` 计算。
- $\mathbf{P}$：投影矩阵（透视或正交）。

### 3.4 投影变换

- 透视投影：符合人眼视觉，远小近大。
- 正交投影：保持尺寸比例，常用于工程视图。
- 本实验支持运行时切换，用于对比两种投影对同一场景的视觉差异。

### 3.5 组合变换与顺序

本实验重点验证：矩阵乘法不满足交换律，故

$$
\mathbf{T}\mathbf{R} \neq \mathbf{R}\mathbf{T}, \quad \mathbf{T}\mathbf{R}\mathbf{S} \neq \mathbf{R}\mathbf{T}\mathbf{S}
$$

在二维和三维场景中分别展示了不同顺序组合带来的显著差异。

## 4. 实验内容与实现过程

### 4.1 工程化

目录结构说明：

- `src/app`：程序入口、窗口主循环、场景调度。
- `src/core`：Shader 封装（读取、编译、链接、uniform 设置）。
- `src/math`：矩阵/向量运算（`perspective`、`ortho`、`lookAt` 等）。
- `src/input`：输入回调封装与按键事件转发。
- `src/scene`：`Scene` 抽象与 `Demo2D`、`Demo3D` 两类场景。
- `shaders`：2D/3D 顶点与片元着色器。
- `docs`：构建日志、截图、文档素材。

### 4.2 二维功能实现（对应要求 1-3）

1. 绘制基础二维图形。
2. 实现二维平移、旋转、缩放、反射、切变。
3. 实现二维组合变换顺序切换（例如 TR 与 RT 对比）。

### 4.3 三维功能实现（对应要求 4-6）

1. 绘制三维立方体，开启深度测试。
2. 实现三维平移、绕 X/Y/Z 轴旋转、缩放。
3. 实现三维组合变换顺序切换（TRS 与 RTS 对比）。

### 4.4 交互与可用性（对应要求 7）

1. 键盘切换 2D/3D 场景与投影模式。
2. 支持相机移动与模型参数动态控制。
3. 增加按键轮询兜底，提升在不同输入环境下的稳定性。
4. 在窗口标题和控制台输出关键状态，便于演示与排障。

### 4.5 自动化验证

程序支持 `--validate` 模式，自动执行检查流程并导出截图到 `docs/screenshots/`，用于功能留痕与报告验收。

## 6. 问题与修复记录

1. 构建工具链不可用：最初无法编译，后完成 MSVC + SDK 环境注入并成功构建。
2. GLAD/GLFW 头文件冲突：统一包含顺序并清理重复宏后恢复正常。
3. `unique_ptr<Demo3D>` 不完整类型析构问题：采用更稳妥的析构实现方式解决。
4. 3D 场景不可见：排查后确认透视矩阵关键项填充位置错误，修复后恢复显示。
5. 交互体验过快/按键不稳定：降低步进、加范围限制、增加键盘轮询兜底。

## 8. 六名组员心得（有感悟）

### 成员A（组长）

这次最深的体会是“图形程序的难点不只在算法，还在系统组织”。早期代码堆在一个文件里时，任何小改动都会牵一发而动全身。重构成模块后，协作效率明显提升。作为组长，我学到的是：先把边界定义清楚，再追求功能速度，项目才不会越做越乱。

### 成员B（数学模块）

我过去对“矩阵对了就能出图”这句话理解不深，这次在透视矩阵排错时真正意识到，哪怕一个元素放错位置，视觉结果都会完全失真。图形学的数学不是纸面推导，而是会直接决定屏幕上每一个像素。以后我会更重视公式与代码的一一对应，而不是“差不多能跑”。

### 成员C（2D 模块）

在做二维组合变换时，我最初直觉认为先旋转还是先平移差别不大，实际演示后才发现两者结果截然不同。这让我对“变换顺序就是语义”有了直观认知。以前把它当作考试知识点，现在能把它看成图形动画和交互设计的基础规则。

### 成员D（3D 模块）

三维部分最让我受益的是“相机思维”的建立。过去我习惯直接动模型，后来逐渐理解很多“看起来物体在动”的效果，本质上是视图矩阵在变化。把模型坐标、世界坐标、观察坐标分开思考后，调试效率提高很多，也更能解释为什么某些角度下物体会消失或变形。

### 成员E（工程与验证）

我负责构建和验证后，明显感受到工程化的重要性。没有自动验证时，大家都说“我这边是好的”，但合并后问题频出；加入 `--validate` 和截图留痕后，讨论从“感觉正确”变成“证据驱动”。这次让我明白，稳定交付并不靠口头保证，而靠可复现流程。

### 成员F（文档与汇报）

写报告过程中，我发现“把技术讲清楚”本身就是一次二次学习。尤其在整理问题修复时，很多同学原本模糊的理解会被追问到非常具体：到底是矩阵顺序错了，还是输入链路断了。文档不是收尾工作，而是帮助团队沉淀方法、减少重复踩坑的核心资产。

## 9. 实验总结

本次实验不仅完成了课程要求的 7 项功能，更实现了从“能跑”到“可维护、可验证、可展示”的升级。团队在实践中形成了三点共识：

1. 图形学的核心是数学与工程并重，公式正确只是起点，工程组织决定上限。
2. 组合变换、视图变换、投影变换必须通过可视化对比来理解，纸面记忆不够。
3. 自动化验证与日志留痕能显著提升团队协作质量。
